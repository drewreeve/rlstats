# Do not type-discriminate IdentityResolver's None return

`IdentityResolver.resolve_car()` (and `resolve_pri`, `resolve_component`) returns `None` for three distinct reasons: the car hasn't been linked to a PRI yet, the PRI hasn't been assigned a UniqueId yet, or the identity is genuinely unknowable (bot, spectator, early disconnect). These states are indistinguishable at the call site, but every handler behaves correctly for all three — the right action is always "skip this frame update." A typed discriminated result (`Resolved | PendingIdentity | Unknown`) was considered and rejected: it would add verbosity to 8 handler call sites for a failure mode that has never been observed in practice. The silent failure risk (a tracked player's PRI never receiving a UniqueId) is real in theory but has not occurred; if it ever does, a finalize-time assertion would be the right place to catch it, not per-frame handler code.
